# search-for-medicine 収益化実装工程・技術調査レポート

本レポートでは、提案した収益化アプローチ（スマホアプリ化、Web Stripe決済、B2Bパスコード）の3パターンについて、具体的な技術要素、実装工程、および想定される作業時間（人時）をエンジニアリング視点で比較・概算します。

プロジェクトは現在静的なWebアプリ（Cloudflare Pages等でのホスティングを想定）であるため、可能な限り**自前のバックエンドサーバーを持たず（サーバーレス/BaaS活用）、かつ個人情報を取得しない**アーキテクチャを選定しています。

---

## パターンA: スマホアプリ化 (Google Play インアプリ課金)

現在のWebアプリをネイティブアプリのガワで包み（Capacitor等）、Google Playストアで配信。課金処理とレシート検証には、匿名IDで利用可能なBaaSである「RevenueCat」を利用します。

* **推奨技術スタック:** Capacitor (Webアプリのネイティブ化), RevenueCat (課金・レシート検証BaaS), Google Play Billing Library
* **個人情報の取り扱い:** 完全に匿名（RevenueCatが自動生成するApp User IDとGoogle側のレシート情報のみでセキュアに判定）。

### 工程表と想定時間 (合計: 約52時間 / 約1〜1.5週間)

| 工程 | 具体的な作業内容 | 想定時間 |
| :--- | :--- | :--- |
| **1. アプリ化環境構築** | Capacitorのプロジェクトへの導入、Android/iOSプロジェクトの生成、アイコン・スプラッシュ画面の設定等のビルド環境構築。 | 8時間 |
| **2. ストア準備** | Google Play Developerアカウントの準備、アプリの初期登録、アプリ内アイテム（サブスクリプションプラン）の作成。（※審査・申請の待ち時間は除く） | 4時間 |
| **3. RevenueCat設定** | RevenueCatアカウント作成、Google Play APIとの連携設定、Entitlements（権限：個別/包括プラン）の紐づけ設定。 | 4時間 |
| **4. 課金UI・購入処理実装** | Capacitor用RevenueCatプラグインの導入。ペイウォール（課金画面）のUI実装と、購入ボタン押下時のネイティブ処理呼び出し。 | 16時間 |
| **5. 権限管理と復元** | アプリ起動時にRevenueCatから購入ステータスを取得し、Pro機能をアンロックするロジック（ローカルストレージとメモリでの状態管理）。「購入の復元」機能の実装。 | 8時間 |
| **6. テスト・審査対応** | Google Playの内部テストトラックでの決済テスト、予期せぬエラー（通信切断等）のハンドリング対応、本番審査提出前の調整。 | 12時間 |

## パターンB: Webアプリのまま (Stripe Checkout)

Webブラウザ上で完結させるため、Stripeを利用します。フロントエンド（Web）から直接Stripeを呼ぶとシークレットキーが漏洩するため、中間層として**Cloudflare Workers**（サーバーレス関数）を利用して安全に決済セッションを発行します。

* **推奨技術スタック:** Stripe Billing / Checkout, Cloudflare Workers (エッジ関数), LocalStorage
* **個人情報の取り扱い:** Stripe側でメールアドレス等を保持。自社システム(Cloudflare)には個人情報を残さず、匿名IDとStripeの顧客IDだけを紐付ける。

### 工程表と想定時間 (合計: 約68時間 / 約1.5〜2週間)

| 工程 | 具体的な作業内容 | 想定時間 |
| :--- | :--- | :--- |
| **1. Stripe準備** | Stripeアカウント開設、本番/テスト環境の準備、Product（個別・包括サブスクプラン）と価格の登録。 | 4時間 |
| **2. CF Workers構築** | Cloudflare Workersのセットアップ。Stripe Secret Keyを環境変数に安全に登録し、中間APIの基盤を作成。 | 8時間 |
| **3. Checkout処理実装** | フロントエンドで匿名IDを生成 → Workerに送信 → WorkerがStripe Checkoutセッションを作成してURLを返す → ユーザーをStripe決済画面へリダイレクト。 | 16時間 |
| **4. サブスク状態の同期** | 決済成功後のコールバック（帰還URL）処理。Local Storageに「有効化フラグ」を保存する仕組みの構築。 | 8時間 |
| **5. Webhook実装** | ユーザーが翌月に課金を停止（解約）した場合など、サブスクの更新状態を検知するためのStripe WebhookエンドポイントをWorkerに実装し、フラグを制御する。 | 16時間 |
| **6. 課金UI・ポータル** | ブラウザ上でのペイウォールUIの実装。ユーザーが自身でプラン変更・解約を行うための「Stripe Customer Portal」への導線実装。 | 8時間 |
| **7. 結合テスト** | Stripeのテストカードを用いた購入、継続、解約、期限切れなどのライフサイクル全般のテスト。 | 8時間 |

## パターンC: B2B 法人向けパスコード決済

薬局等と直接契約（請求書払い等）し、専用のパスコードを発行するアナログかつ最も確実なモデルです。システムの役割は「パスコードの有効性検証」のみとなります。

* **推奨技術スタック:** Cloudflare Workers, Cloudflare D1(SQLite) または KV (Key-Valueストア)
* **個人情報の取り扱い:** 完全匿名。施設に紐づくランダムな文字列（コード）のみ。

### 工程表と想定時間 (合計: 約32時間 / 約4日〜1週間)

| 工程 | 具体的な作業内容 | 想定時間 |
| :--- | :--- | :--- |
| **1. DB・ストレージ設計** | 有効なパスコードと期限を管理するための軽量DB（Cloudflare D1等）のスキーマ定義と初期構築。（管理用スクリプトの作成含む） | 8時間 |
| **2. 検証用API作成** | フロントから送信されたパスコードを受け取り、DBと照合して「有効・無効」「対象プラン（包括か個別か）」を返すCloudflare Worker APIの実装。 | 8時間 |
| **3. 認証UI・ロジック** | ユーザー向けのパスコード入力モーダルの設計・実装。APIへの通信処理とエラーハンドリング（無効なコード、期限切れ等）。 | 8時間 |
| **4. 権限アンロック処理** | 認証成功時に返却されるトークン（JWT等）をローカルストレージに保存し、アプリ起動時に有効性をチェックしてPro機能を解放するロジックの実装。 | 8時間 |

## 総括と技術的知見

1. **実装の容易さ（早さ）:** `パターンC (パスコード) > パターンA (アプリ化) > パターンB (Web Stripe)`
   * パターンB(Stripe)は、課金の「継続」「解約」「期限切れ」のライフサイクルの状態管理をWebhookを使って自前で組む必要があり、セキュアな実装に想定以上の工数がかかります。
   * パターンA(アプリ化)は、RevenueCatという外部サービスが状態管理の複雑さを吸収してくれるため、ネイティブ化の壁さえ越えれば実装は素直です。
2. **保守運用の観点:**
   * アプリ化（パターンA）はストアの規約変更やOSアップデートに追従するメンテナンスコストが永続的に発生します。
   * Webのまま（パターンB, C）は、一度動くセキュアなAPI（Workers）を作ってしまえば、メンテナンスコストは極めて低く済みます。

これらを踏まえ、まずは最速で付加価値を提供できる**パターンC（法人パスコードモデル）**から着手して施設利用実績を作り、並行して一般ユーザー向けに**パターンA（Capacitor+RevenueCatによるアプリ化）**の技術検証（PoC）を進めるアプローチが、リスクを抑えつつ収益化を目指す上で効果的と考えられます。
