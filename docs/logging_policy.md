# ログ設計指針 (Logging Policy)

## 1. ログをとる目的

`search-for-medicine` プロジェクトにおいてログを記録する主な目的は以下の通りです。

1.  **デバッグとトラブルシューティング**: 
    - ユーザーの環境（ブラウザ、通信状況）に依存するエラーの特定。
    - Google Sheets や外部 API（ヒヤリハット、花粉等）との連携失敗の検知。
2.  **ユーザー行動の把握（アナリティクス）**: 
    - どの医薬品が頻繁に検索されているか（需要の把握）。
    - どのサブアプリ（小児用計算機、レシピ提案等）が使われているか。
3.  **パフォーマンス監視**: 
    - 初回読み込み時間や、`localForage` によるキャッシュの効果測定。
4.  **セキュリティと監査**: 
    - 異常な頻度でのアクセス（スクレイピング等）の検知。

## 2. メリット

-   **再現困難な不具合の解消**: 開発環境では再現しないエラーの背景（エラーメッセージ、スタックトレース）を取得できる。
-   **サービス改善の根拠**: 「どの機能に注力すべきか」をデータに基づいて判断できる。
-   **ユーザーサポートの迅速化**: 問い合わせがあった際、発生状況を正確に把握できる。

## 3. デメリットとリスク

-   **プライバシー・セキュリティリスク**: 
    - 検索ワードや入力内容に個人情報（PII）が含まれる場合、それをログに含めてしまうリスク。
-   **パフォーマンスへの影響**: 
    - ログの送信処理がメインスレッドをブロックしたり、通信量を増やしたりする可能性がある（特にモバイル環境）。
-   **ストレージ・運用コスト**: 
    - ログを保存するサーバーの利用料や、ログを監視・分析するための工数。

## 4. search-for-medicine におけるログ戦略（提案）

現状の静的サイト構成を維持しつつ、以下のステップでの導入を提案します。

| レベル | 内容 | 推奨されるツール |
| :--- | :--- | :--- |
| **Error** | クラッシュや API 失敗の記録 | Sentry (無料枠内) または GAS 経由の通知 |
| **Usage** | 検索語句、ボタンクリック等の行動ログ | Google Analytics 4 (GA4) のカスタムイベント |
| **Audit** | 管理操作（もしあれば）の記録 | バックエンド（Cloudflare Workers 等）でのログ出力 |

### 5. コンプライアンスと注意点

- **情報の選別**: 氏名や住所などの個人情報は絶対にログに含めない。
- **プライバシーポリシー**: ログを収集する場合は、プライバシーポリシー（`privacy.html`）を更新し、利用者に通知する。
